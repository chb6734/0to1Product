## 1. 제품 개요

### 비전과 목표

**비전**: "음악으로 모인 사람들이, 음악으로 살아갈 수 있도록"

**MVP 목표**: 플랫폼 중립적 음악 편지 서비스를 통해 핵심 가치를 검증한다.

### 핵심 가치 제안

1. **플랫폼 중립성**: 어떤 음악 스트리밍 서비스를 사용하든 음악을 공유할 수 있다
2. **감성 전달**: 단순 링크가 아닌 메시지와 함께 음악을 전달한다
3. **영구 보관**: 받은 편지를 영구적으로 보관하고 다시 찾을 수 있다

### 타겟 사용자

#### 1차 타겟: 일반 팬 & 음악 애호가 (20-35세)

- **페르소나**: 음악을 좋아하고 친구/연인에게 추천하고 싶은 사람
- **니즈**:
  - 플랫폼이 달라도 음악을 공유하고 싶음
  - 의미 있는 방식으로 음악을 선물하고 싶음
- **사용 시나리오**: 생일 선물, 위로, 응원, 기념일

#### 2차 타겟: 아티스트 & 음악인

- **페르소나**: 인디/신인 아티스트, 세션 뮤지션
- **니즈**:
  - 팬과 소통하고 싶음
  - 셋리스트를 효과적으로 공유하고 싶음
- **사용 시나리오**: 페스티벌 셋리스트 공유, 신곡 소개

## 2. 기능 명세

### 우선순위 정의

- **P0 (필수)**: MVP의 핵심 가치를 검증하기 위해 반드시 필요

### P0: 핵심 기능

#### 2.1 사용자 인증 및 프로필

**기능 설명**:

- 소셜 로그인을 통한 간편 가입
- 기본 프로필 설정

**상세 요구사항**:

- 소셜 로그인 지원: Google, Kakao, Apple
- 닉네임 설정 (필수)
- 프로필 이미지 업로드 (선택)
- 이메일 주소 (소셜 로그인에서 자동 수집)

**사용자 시나리오**:

1. 사용자가 "시작하기" 버튼 클릭
2. 소셜 로그인 선택 (Google/Kakao/Apple)
3. 권한 승인 후 자동 가입 완료
4. 닉네임 입력 화면으로 이동
5. 프로필 이미지 선택 (선택적)
6. 메인 화면으로 이동

**사용자 변화**:

- **Before**: 음악 공유를 위해 여러 앱을 설치하고 계정을 만들어야 함
- **After**: 한 번의 로그인으로 모든 플랫폼과 연동 가능
- **변화 지표**: 가입 완료 시간 30초 이내, 가입 전환율 70% 이상

**레버리지 포인트 활용**:

- 간편한 가입으로 음악인 커뮤니티 진입 장벽 낮춤
- 사용자 데이터 수집으로 신뢰 시스템 기반 마련

**성공 기준**:

- 가입 완료까지 30초 이내
- 가입 전환율 70% 이상

#### 2.2 음악 편지 생성

**기능 설명**:

- 플레이리스트를 만들고 메시지를 추가하여 음악 편지를 생성

**상세 요구사항**:

**2.2.1 곡 검색 및 추가**

- 곡 검색 (제목, 아티스트명)
- 검색 결과 표시 (제목, 아티스트, 앨범 커버)
- 플레이리스트에 곡 추가
- 곡 삭제
- 곡 순서 변경 (드래그 앤 드롭)

**2.2.2 메시지 작성**

- 메시지 입력 (텍스트, 최대 500자)
- AI 메시지 생성 (선택적)
  - 템플릿 기반 (생일, 위로, 축하, 사랑, 우정)
  - 사용자가 수정 가능
- 각 곡마다 메모 추가 (선택적, 최대 100자)

**2.2.3 편지 미리보기**

- 카드 형태로 미리보기
- 모바일/데스크톱 반응형

**사용자 시나리오**:

1. "편지 만들기" 버튼 클릭
2. 곡 검색 및 추가 (최소 1곡)
3. 메시지 작성 또는 AI 생성 선택
4. 각 곡에 메모 추가 (선택적)
5. 미리보기 확인
6. "편지 완성" 클릭

**사용자 변화**:

- **Before**: 음악을 단순 링크로 공유, 맥락 없음
- **After**: 메시지와 함께 음악을 "선물"로 인식, 감성적 가치 부여
- **변화 지표**: 편지 생성 완료 시간 3분 이내, 사용자 만족도 4.0/5.0 이상

**레버리지 포인트 활용**:

- 플랫폼 중립성의 핵심 구현: 어떤 플랫폼의 곡이든 검색 가능
- 플레이리스트 활동 기록으로 신뢰 시스템 기반 구축

**성공 기준**:

- 편지 생성 완료까지 3분 이내
- 사용자 만족도 4.0/5.0 이상

#### 2.3 플랫폼 연동

**기능 설명**:

- 여러 음악 스트리밍 플랫폼과 연동하여 사용자의 플랫폼에 플레이리스트 저장

**상세 요구사항**:

**2.3.1 지원 플랫폼**

- Spotify (OAuth 2.0)
- YouTube Music (OAuth 2.0)
- Apple Music (MusicKit JS)
- 멜론 (사용자 수동 입력 - API 부재)

**2.3.2 곡 매칭 알고리즘**

- 곡 제목 + 아티스트명으로 매칭
- 매칭 실패 시 사용자에게 알림
- 대안 곡 제시 (선택적)

**2.3.3 플레이리스트 저장**

- 사용자가 선택한 플랫폼에 플레이리스트 생성
- 원클릭 저장
- 저장 성공/실패 피드백

**사용자 시나리오**:

1. 편지를 받은 사용자가 링크 클릭
2. 편지 열람
3. "내 플랫폼에 저장" 버튼 클릭
4. 플랫폼 선택 (Spotify/YouTube/Apple Music)
5. 로그인 (처음 사용 시)
6. 플레이리스트 저장 완료

**사용자 변화**:

- **Before**: 플랫폼이 다르면 음악 공유 불가, 링크만 받고 끝
- **After**: 플랫폼 상관없이 음악을 받아 자신의 플랫폼에 저장, 음악을 "경험"으로 인식
- **변화 지표**: 플레이리스트 저장 성공률 60% 이상, 저장 완료 시간 10초 이내

**레버리지 포인트 활용**:

- **핵심 레버리지**: 플랫폼 중립성의 완전한 구현
- 마케팅 메시지의 핵심: "어떤 플랫폼을 쓰든 음악을 공유하세요"
- 경쟁자 모방 어려움: 기술적 진입장벽

**성공 기준**:

- 플레이리스트 저장 성공률 60% 이상
- 저장 완료까지 10초 이내

#### 2.4 편지 전송 및 수신

**기능 설명**:

- 편지를 생성한 후 수신자에게 전송하고, 받은 편지를 관리

**상세 요구사항**:

**2.4.1 편지 전송**

- 수신자 이메일 입력
- 또는 링크 공유 (카카오톡, 문자, SNS)
- 전송 완료 알림

**2.4.2 편지 열람**

- 링크를 통한 편지 열람 (로그인 불필요)
- 카드 형태로 표시
- 모바일 최적화

**2.4.3 편지 보관함**

- 받은 편지 보관함
- 보낸 편지 보관함
- 날짜별 정렬
- 검색 기능 (보낸 사람, 곡 제목)

**사용자 시나리오**:

1. 편지 생성 완료 후 "전송하기" 클릭
2. 수신자 이메일 입력 또는 링크 복사
3. 전송 완료
4. 수신자가 링크 클릭하여 편지 열람
5. 받은 편지는 자동으로 보관함에 저장 (로그인 시)

**사용자 변화**:

- **Before**: 음악 링크는 대화방에서 묻힘, 다시 찾기 어려움
- **After**: 받은 편지를 영구 보관, 날짜별/사람별로 정리, 추억으로 인식
- **변화 지표**: 편지 열람률 70% 이상, 편지 재방문률 30% 이상

**레버리지 포인트 활용**:

- 영구 보관으로 사용자 리텐션 향상
- 바이럴 메커니즘: 편지를 받은 사람이 자연스럽게 사용자로 전환

**성공 기준**:

- 편지 열람률 70% 이상
- 편지 재방문률 30% 이상

## 3. 기술 요구사항

### 기술 스택

#### Frontend

- **Framework**: Next.js 14 (App Router)
- **Language**: TypeScript
- **Styling**: Tailwind CSS
- **UI Components**: shadcn/ui 또는 Tailwind UI
- **State Management**: React Context 또는 Zustand
- **Form Handling**: React Hook Form

#### Backend

- **Database**: Supabase (PostgreSQL)
- **Authentication**: Supabase Auth
- **Storage**: Supabase Storage (프로필 이미지)
- **API**: Next.js API Routes

#### External APIs

- **Spotify**: Spotify Web API
- **YouTube Music**: YouTube Data API v3
- **Apple Music**: MusicKit JS
- **AI**: OpenAI API (GPT-4o Mini) - 선택적

#### 배포 및 인프라

- **Hosting**: Vercel
- **Database**: Supabase (무료 플랜)
- **CDN**: Vercel Edge Network
- **Monitoring**: Vercel Analytics

### 성능 요구사항

- **페이지 로딩 시간**: 2초 이하
- **API 응답 시간**: 1초 이하
- **에러율**: 1% 이하
- **가용성**: 99% 이상

### 보안 요구사항

- HTTPS 필수
- 소셜 로그인 OAuth 2.0
- 사용자 데이터 암호화
- API 키 환경 변수 관리
- SQL Injection 방지
- XSS 방지

### 확장성 요구사항

- 초기: 1,000명 사용자 지원
- 6개월: 10,000명 사용자 지원
- 서버리스 아키텍처로 자동 스케일링

### 3.4 스케일링 전략 상세

#### API 할당량 관리 전략

**문제**: 사용자 증가 시 API 호출 급증, 할당량 초과 위험

**전략**:

1. **사용자별 API 키 활용 (Spotify)**

   - 각 사용자가 자신의 Spotify API 키 사용
   - 서버 부하 분산
   - 할당량 문제 해결
   - 구현: OAuth 2.0으로 사용자 토큰 획득

2. **캐싱 전략**

   - 곡 정보 캐싱 (Redis 또는 Supabase Cache)
   - 검색 결과 캐싱 (5분 TTL)
   - 플레이리스트 메타데이터 캐싱 (1시간 TTL)
   - 예상 효과: API 호출 70% 감소

3. **API 호출 최적화**

   - 배치 처리: 여러 곡을 한 번에 검색
   - 요청 큐잉: 동시 요청 제한
   - 재시도 로직: Exponential backoff
   - 예상 효과: API 호출 30% 감소

4. **모니터링 및 알림 시스템**
   - API 호출량 실시간 모니터링
   - 할당량 80% 도달 시 알림
   - 자동 스케일링 트리거
   - 대시보드: Vercel Analytics + Supabase Dashboard

**목표**: 사용자 10,000명까지 API 할당량 문제 없이 운영

---

#### 데이터베이스 최적화 전략

**문제**: 사용자 증가 시 쿼리 성능 저하

**전략**:

1. **인덱싱 전략**

   - 편지 ID (Primary Key)
   - 사용자 ID (Foreign Key)
   - 생성 날짜 (정렬용)
   - 수신자 이메일 (검색용)
   - 예상 효과: 쿼리 속도 10배 향상

2. **쿼리 최적화**

   - N+1 쿼리 방지 (JOIN 사용)
   - 페이지네이션 (20개씩)
   - 필요한 컬럼만 SELECT
   - 예상 효과: 쿼리 시간 50% 감소

3. **읽기/쓰기 분리 (필요 시)**

   - 읽기 전용 복제본 사용
   - Supabase Read Replicas 활용
   - 예상 효과: 읽기 성능 2배 향상

4. **캐싱 전략**
   - Redis 또는 Supabase Cache
   - 편지 목록 캐싱 (5분 TTL)
   - 사용자 프로필 캐싱 (10분 TTL)
   - 예상 효과: 데이터베이스 부하 60% 감소

**목표**: 사용자 10,000명까지 쿼리 응답 시간 1초 이하 유지

---

#### 에러 핸들링 전략

**문제**: 플랫폼 API 오류, 네트워크 오류, 토큰 만료 등

**전략**:

1. **Exponential Backoff 패턴**

   - 첫 재시도: 1초 후
   - 두 번째 재시도: 2초 후
   - 세 번째 재시도: 4초 후
   - 최대 3회 재시도
   - 예상 효과: 일시적 오류 90% 해결

2. **Circuit Breaker 패턴**

   - API 실패율 50% 초과 시 5분간 차단
   - 자동 복구 시도
   - 사용자에게 명확한 에러 메시지
   - 예상 효과: 연쇄 실패 방지

3. **재시도 로직 상세화**

   - 네트워크 오류: 즉시 재시도
   - API 오류 (4xx): 재시도 안 함
   - API 오류 (5xx): Exponential backoff로 재시도
   - 토큰 만료: 자동 갱신 후 재시도

4. **사용자 피드백 메커니즘**
   - 명확한 에러 메시지
   - 재시도 버튼 제공
   - 대안 플랫폼 제안
   - 고객 지원 채널 안내

**목표**: 에러율 1% 이하 유지, 사용자 만족도 유지
